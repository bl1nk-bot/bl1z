## หลักเกณฑ์การปฏิบัติตน (Code of Conduct)

โปรเจกต์นี้ปฏิบัติตามหลักเกณฑ์การปฏิบัติตนเพื่อให้มั่นใจว่าสภาพแวดล้อมในการพัฒนานั้นเป็นมิตรต่อทุกคน ในการเข้าร่วมโปรเจกต์นี้ คุณตกลงที่จะ:

- ให้ความเคารพและยอมรับความแตกต่าง
- มุ่งเน้นไปที่การให้ข้อมูลย้อนกลับเชิงสร้างสรรค์
- ยอมรับความรับผิดชอบต่อความผิดพลาด
- แสดงความเห็นอกเห็นใจต่อผู้มีส่วนร่วมคนอื่น
- ช่วยสร้างชุมชนเชิงบวก

## การเริ่มต้นใช้งาน (Getting Started)

### สิ่งที่จำเป็นต้องมี (Prerequisites)

- **Rust**: เวอร์ชัน 1.70 ขึ้นไป (ตรวจสอบด้วยคำสั่ง `rustc --version`)
- **Cargo**: มาพร้อมกับ Rust (ตรวจสอบด้วยคำสั่ง `cargo --version`)
- **Git**: สำหรับการควบคุมเวอร์ชัน

### การตั้งค่าอย่างรวดเร็ว (Quick Setup)

1. **ทำซ้ำ (Fork) และคัดลอก (Clone)** repository:
   ```bash
   git clone https://github.com/billlzzz26/bl1z.git
   cd bl1z
   ```

2. **ตั้งค่าสภาพแวดล้อมการพัฒนา**:
   ```bash
   cargo check  # ตรวจสอบว่าคอมไพล์ผ่านหรือไม่
   cargo test   # รันชุดการทดสอบ
   ```

## การตั้งค่าสภาพแวดล้อมสำหรับการพัฒนา (Development Setup)

### การติดตั้งสิ่งที่จำเป็น (Dependencies)

สำหรับ Ubuntu/Debian:
```bash
sudo apt update
sudo apt install -y build-essential
```

สำหรับ macOS:
```bash
xcode-select --install
```

สำหรับ Windows: ติดตั้ง [Build Tools for Visual Studio](https://visualstudio.microsoft.com/downloads/) หรือใช้งานผ่าน WSL

### เครื่องมือสำหรับการพัฒนา (Development Tools)

```bash
# ติดตั้งเครื่องมือเพิ่มเติม
cargo install cargo-watch    # สำหรับการเฝ้าดูการเปลี่ยนแปลงไฟล์
cargo install cargo-edit     # สำหรับการจัดการ dependencies
cargo install cargo-audit    # สำหรับการตรวจสอบความปลอดภัย
```

### การตั้งค่า IDE (IDE Setup)

**VS Code** (แนะนำ):
- ติดตั้งส่วนขยาย Rust Analyzer
- ติดตั้ง "CodeLLDB" สำหรับการทำ Debugging
- ติดตั้ง "crates" สำหรับการจัดการ dependencies

**บรรณาธิการข้อความอื่น ๆ**:
- บรรณาธิการข้อความใด ๆ ที่รองรับภาษา Rust
- Rust Analyzer LSP จะให้ฟีเจอร์การพัฒนาที่ยอดเยี่ยม

## ขั้นตอนการทำงาน (Development Workflow)

### 1. เลือกประเด็นปัญหา (Issue)

- ตรวจสอบ [GitHub Issues](https://github.com/bl1nk-bot/bl1z/issues) สำหรับงานที่ยังค้างอยู่
- ค้นหาประเด็นปัญหาที่ติดป้ายกำกับ `good first issue` หรือ `help wanted`
- เขียนคอมเมนต์ในประเด็นปัญหาเพื่อระบุว่าคุณกำลังดำเนินการอยู่

### 2. สร้างกิ่งงาน (Branch)

```bash
git checkout -b feature/my-feature
# หรือ
git checkout -b fix/issue-number-description
```

### 3. ดำเนินการแก้ไข

- เขียนข้อความ commit ที่ชัดเจนและตรงประเด็น
- ปฏิบัติตามมาตรฐานการเขียนโค้ดด้านล่าง
- เพิ่มการทดสอบสำหรับฟังก์ชันการทำงานใหม่
- อัปเดตเอกสารประกอบตามความจำเป็น

### 4. ตรวจสอบความถูกต้อง

```bash
cargo check                    # ตรวจสอบระบบชนิดข้อมูล
cargo fmt --all -- --check     # จัดรูปแบบโค้ด
cargo clippy                   # ตรวจสอบคุณภาพโค้ด (Linting)
cargo test --verbose           # รันการทดสอบ
cargo bench                    # รัน benchmarks (หากมี)
```

### 5. ส่งคำร้องขอรวมโค้ด (Pull Request)

- ผลัก (Push) กิ่งงานของคุณไปยัง GitHub
- สร้าง Pull Request พร้อมคำอธิบายที่ชัดเจน
- อ้างอิงประเด็นปัญหาที่เกี่ยวข้อง
- ร้องขอการรีวิวจากผู้ดูแลระบบ

## มาตรฐานการเขียนโค้ด (Coding Standards)

### รูปแบบภาษา Rust (Rust Style)

- ปฏิบัติตาม [Rust Style Guide](https://doc.rust-lang.org/style-guide/) อย่างเป็นทางการ
- ใช้งาน `rustfmt` สำหรับการจัดรูปแบบโค้ดอัตโนมัติ
- ใช้งาน `clippy` สำหรับการตรวจสอบคุณภาพโค้ด (ต้องผ่านคำเตือนทั้งหมด)

### การจัดการโครงสร้างโค้ด (Code Organization)

- **โมดูล (Modules)**: แยกฟังก์ชันการทำงานตามหลักการทางตรรกะ
- **ฟังก์ชัน (Functions)**: มีขนาดเล็ก มุ่งเน้นงานเดียว และมีความรับผิดชอบเดียว
- **ชนิดข้อมูล (Types)**: ตั้งชื่อให้ชัดเจน และมีเอกสารประกอบที่ครบถ้วน
- **การจัดการข้อผิดพลาด (Error Handling)**: ใช้งานชนิดข้อมูล `Result` และข้อความแสดงข้อผิดพลาดที่มีความหมาย

### เอกสารประกอบ (Documentation)

- **Public APIs**: ต้องมีเอกสารประกอบที่ครอบคลุม
- **ตรรกะที่ซับซ้อน**: ใส่คอมเมนต์อธิบายเหตุผลประกอบ
- **ตัวอย่าง (Examples)**: ให้ตัวอย่างโค้ดในจุดที่เป็นประโยชน์

### มาตรฐานการตั้งชื่อ (Naming Conventions)

- **ฟังก์ชัน**: `snake_case`
- **ชนิดข้อมูล/Enums**: `PascalCase`
- **ค่าคงที่**: `SCREAMING_SNAKE_CASE`
- **โมดูล**: `snake_case`

### ตัวอย่างโครงสร้างโค้ด

```rust
/// คำอธิบายสั้น ๆ เกี่ยวกับหน้าที่ของฟังก์ชันนี้
///
/// # Arguments
/// * `param` - คำอธิบายพารามิเตอร์
///
/// # Returns
/// คำอธิบายค่าที่ส่งกลับ
///
/// # Examples
/// ```
/// use bl1z::example_function;
/// let result = example_function(42);
/// assert_eq!(result, 84);
/// ```
pub fn example_function(param: i32) -> i32 {
    param * 2
}
```

## การทดสอบ (Testing)

### การรันการทดสอบ

```bash
# การทดสอบทั้งหมด
cargo test

# การทดสอบเฉพาะรายการ
cargo test test_name

# การทดสอบพร้อมแสดงผลลัพธ์
cargo test -- --nocapture

# การทดสอบในเอกสารประกอบ (Doc tests)
cargo test --doc
```

### การเขียนการทดสอบ

- **การทดสอบระดับหน่วย (Unit Tests)**: ทดสอบฟังก์ชันหรือองค์ประกอบย่อย
- **การทดสอบระดับรวม (Integration Tests)**: ทดสอบฟังก์ชันการทำงานขนาดใหญ่
- **การทดสอบในเอกสารประกอบ (Doc Tests)**: ตัวอย่างในเอกสารประกอบ
- **การทดสอบคุณสมบัติ (Property Tests)**: ใช้งานไลบรารีอย่าง `proptest` สำหรับสถานการณ์ที่ซับซ้อน

### ความครอบคลุมของการทดสอบ (Test Coverage)

- มุ่งเป้าไปที่ความครอบคลุมของการทดสอบที่สูง (>80%)
- ทดสอบทั้งกรณีที่สำเร็จและกรณีที่เกิดข้อผิดพลาด
- รวมกรณีขอบเขต (edge cases) และเงื่อนไขวิกฤต (boundary conditions)

### ตัวอย่างโครงสร้างการทดสอบ

```rust
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_basic_functionality() {
        // จัดเตรียม (Arrange)
        let input = 42;
        let expected = 84;

        // ดำเนินการ (Act)
        let result = example_function(input);

        // ตรวจสอบ (Assert)
        assert_eq!(result, expected);
    }

    #[test]
    fn test_error_cases() {
        // ทดสอบการจัดการข้อผิดพลาด
        let result = example_function(-1);
        assert!(result.is_err());
    }
}
```

## การส่งมอบการเปลี่ยนแปลง (Submitting Changes)

### ข้อความ Commit (Commit Messages)

ปฏิบัติตามรูปแบบ conventional commit:

```
type(scope): description

[เนื้อหาทางเลือก]

[ส่วนท้ายทางเลือก]
```

ประเภท (Types):
- `feat`: ฟีเจอร์ใหม่
- `fix`: แก้ไขบั๊ก
- `docs`: เอกสารประกอบ
- `style`: การเปลี่ยนแปลงรูปแบบโค้ด
- `refactor`: การปรับปรุงโครงสร้างโค้ด
- `test`: การทดสอบ
- `chore`: งานซ่อมบำรุง

ตัวอย่าง:
```
feat(parser): add support for array literals
fix(eval): correct division by zero handling
docs(api): update function documentation
```

### แนวทางการส่งคำร้องขอรวมโค้ด (Pull Request Guidelines)

- **หัวข้อ (Title)**: หัวข้อที่ชัดเจนและสื่อความหมาย
- **คำอธิบาย (Description)**: อธิบายว่าทำอะไรและเพราะอะไร
- **การทดสอบ (Tests)**: รวมการทดสอบสำหรับฟังก์ชันการทำงานใหม่
- **เอกสารประกอบ (Documentation)**: อัปเดตเอกสารหากจำเป็น
- **การเปลี่ยนแปลงที่ส่งผลกระทบ (Breaking Changes)**: ระบุและอธิบายให้ชัดเจน
- **ประเด็นปัญหาที่เกี่ยวข้อง**: อ้างอิงหมายเลข issue

### รูปแบบ PR (PR Template)

```markdown
## รายละเอียด (Description)
คำอธิบายสั้น ๆ เกี่ยวกับการเปลี่ยนแปลง

## ประเภทของการเปลี่ยนแปลง (Type of Change)
- [ ] แก้ไขบั๊ก (Bug fix)
- [ ] ฟีเจอร์ใหม่ (New feature)
- [ ] การเปลี่ยนแปลงที่ส่งผลกระทบ (Breaking change)
- [ ] อัปเดตเอกสารประกอบ (Documentation update)

## การทดสอบ (Testing)
- [ ] การทดสอบเดิมทั้งหมดผ่าน
- [ ] เพิ่มการทดสอบใหม่
- [ ] ทำการทดสอบด้วยตนเองแล้ว

## รายการตรวจสอบ (Checklist)
- [ ] โค้ดคอมไพล์ได้โดยไม่มีคำเตือน
- [ ] การทดสอบผ่านทั้งหมด
- [ ] อัปเดตเอกสารประกอบแล้ว
- [ ] ข้อความ commit ปฏิบัติตามมาตรฐาน
```

## การรายงานปัญหา (Reporting Issues)

### การรายงานบั๊ก (Bug Reports)

กรุณาระบุข้อมูลต่อไปนี้:
- **คำอธิบาย**: อธิบายปัญหาให้ชัดเจน
- **ขั้นตอนการเกิดปัญหา**: ตัวอย่างโค้ดที่สั้นที่สุด
- **พฤติกรรมที่คาดหวัง**: สิ่งที่ควรเกิดขึ้น
- **พฤติกรรมที่เกิดขึ้นจริง**: สิ่งที่เกิดขึ้นจริง
- **สภาพแวดล้อม**: เวอร์ชัน Rust, ระบบปฏิบัติการ เป็นต้น
- **ข้อมูลเพิ่มเติม**: ภาพถ่ายหน้าจอ, บันทึกเหตุการณ์ (logs) เป็นต้น

### การร้องขอฟีเจอร์ใหม่ (Feature Requests)

กรุณาระบุข้อมูลต่อไปนี้:
- **คำอธิบาย**: ฟีเจอร์ที่คุณต้องการ
- **กรณีการใช้งาน**: เพราะเหตุใดถึงจำเป็นต้องมี
- **แนวทางที่เสนอ**: วิธีการทำงานที่คาดไว้
- **ทางเลือกอื่น**: แนวทางอื่นที่ได้พิจารณาแล้ว

### ประเด็นด้านความปลอดภัย (Security Issues)

สำหรับช่องโหว่ด้านความปลอดภัย:
- **ห้าม** สร้าง GitHub issues แบบสาธารณะ
- ส่งอีเมลถึงผู้ดูแลโดยตรง
- ดูรายละเอียดที่ [นโยบายด้านความปลอดภัย (Security Policy)](SECURITY.md)

## การยอมรับผลงาน (Recognition)

ผู้มีส่วนร่วมจะได้รับ:
- การระบุชื่อใน CHANGELOG.md สำหรับการเปลี่ยนแปลงที่สำคัญ
- การระบุชื่อในบันทึกการเปิดตัว (release notes)
- การเพิ่มชื่อในรายการผู้มีส่วนร่วม (หากต้องการ)

ขอขอบคุณที่มีส่วนร่วมกับ bl1z! 🎉
